# TeachHub Billing Policy
**Version:** 2.0 — Post-paid per-teacher B2B  
**Effective:** Phase 2 (billing rework, migration 013)  
**Scope:** TeachHub ↔ center only. TeachHub never touches center ↔ parent/student money.

---

## 1. Model overview

TeachHub charges **centers** (not teachers, not students, not parents) on a **post-paid monthly** basis.

```
TeachHub (platform)
  └─ center  ← the paying customer
       ├─ owner (role='owner')  — never billed
       └─ teacher (role='teacher')  — billable after trial
```

The center receives one invoice per calendar month, generated by the platform admin on **day 1 of month M+1**. There is no automated billing job in Phase 2; generation is manual via the platform admin UI.

---

## 2. Billing unit

**One billable teacher = 1 × center.price_per_teacher for the month.**

Counting rule: a teacher counts as 1 billable teacher for month M if they were active (`active = true`) at **any point** during month M.  
There is **no proration** — being active for one day or thirty days costs the same.

Formally, a teacher is counted for period `[period_start, period_end)` if:
- `role = 'teacher'` — owners are never billed  
- `billable_from IS NOT NULL AND billable_from <= period_end` — trial has ended  
- `(active = true) OR (deactivated_at IS NOT NULL AND deactivated_at >= period_start)`  
  — currently active, OR was deactivated this month (active during at least some of the month)

---

## 3. Trial

- **Duration:** 30 days from a teacher's **first login**.  
- **Mechanics:** On first login (`billable_from IS NULL`), the system writes `billable_from = NOW() + INTERVAL '30 days'` to `admin.billable_from`.  
- **Enforcement:** `SetBillableFrom` executes `UPDATE admin SET billable_from = $2 WHERE id = $1 AND billable_from IS NULL` — the `WHERE billable_from IS NULL` guard means concurrent or repeated calls are no-ops at the DB level.  
- **One-way:** Once `billable_from` is set it is never changed. Deactivating and reactivating a teacher does **not** reset or extend the trial.  
- **Owner accounts** have `billable_from` ignored — they are never billable regardless of its value.  
- A teacher who has never logged in has `billable_from = NULL`. They are not counted in any invoice until they first log in and the trial clock starts.

---

## 4. Currency

- Currency is stored per center in `center.currency` (e.g. `'EUR'`, `'DZD'`).  
- It is seeded at center creation from the center's country using the mapping below.  
- It is **never inferred at runtime** from the country — always read `center.currency`.  
- Platform admin can edit `center.currency` at any time via the center detail page.

| Country | Default currency | Default price_per_teacher |
|---------|-----------------|--------------------------|
| FR      | EUR             | 10                       |
| DZ (all others) | DZD    | 500                      |

`geo.CurrencyForCountry` may be used during center **creation** to seed the default. It must not be used for display at runtime.

## 5. Invoice lifecycle

| Status   | Meaning                                              |
|----------|------------------------------------------------------|
| `unpaid` | Generated, awaiting payment confirmation             |
| `paid`   | Platform admin has recorded payment                  |
| `cancelled` | Platform admin cancelled (error correction)       |

Invoice fields stored in `center_invoice`:

| Field             | Type            | Notes                                    |
|-------------------|-----------------|------------------------------------------|
| id                | SERIAL PK       |                                          |
| center_id         | INT FK → center |                                          |
| period_month      | DATE            | Always YYYY-MM-01                        |
| teacher_count     | INT             | Billable teacher count at generation time |
| price_per_teacher | NUMERIC(10,2)   | Snapshot of center.price_per_teacher     |
| currency          | TEXT            | Snapshot of center.currency              |
| total_amount      | NUMERIC(10,2)   | teacher_count × price_per_teacher        |
| status            | TEXT            | unpaid / paid / cancelled                |
| paid_at           | TIMESTAMPTZ     | Nullable                                 |
| paid_method       | TEXT            | cash, ccp, virement, other               |
| paid_reference    | TEXT            | Reference number / receipt               |
| generated_at      | TIMESTAMPTZ     | When the platform admin ran generation   |
| UNIQUE            | (center_id, period_month) | Prevents double-generation   |

---

## 6. Dunning policy

- **No lockout.** An unpaid invoice never blocks teacher or owner login.  
- **Dunning = banner.** If any `center_invoice.status = 'unpaid'` exists for a center, the owner's center dashboard shows a visible warning banner ("Facture impayée — contactez l'équipe TeachHub").  
- **Follow-up** is done manually by the platform admin (phone/email).  
- `subscription_status` on the `center` table is retained as **internal finance metadata** (can be set to `suspended` or `cancelled` by platform admin to record non-payment). It **never gates login** — the login and per-request subscription checks are removed.

---

## 7. Owner accounts

Teachers with `role = 'owner'` are **never billed**:
- Excluded from the `teacher_count` in every `center_invoice`.  
- Their `billable_from` is irrelevant and should be ignored in all billing queries.

---

## 8. Deactivation audit trail

| Column          | Behavior                                               |
|-----------------|--------------------------------------------------------|
| `active`        | Current state flag. `true` = can log in. `false` = blocked. |
| `deactivated_at`| Set to `NOW()` when owner deactivates a teacher. Cleared to `NULL` when owner reactivates. Never accumulates history (single value). |
| `billable_from` | Set once on first login. Never touched on deactivate/reactivate. |

---

## 9. Invoice generation algorithm (Phase 2 — manual)

Trigger: Platform admin POSTs to `/platform/centers/:id/generate-invoices?month=YYYY-MM`.

Steps:
1. Parse `period_start = YYYY-MM-01 00:00:00 UTC`, `period_end = period_start + 1 month`.
2. Count billable teachers:
   ```sql
   SELECT COUNT(*) FROM admin
   WHERE center_id = $1
     AND role = 'teacher'
     AND billable_from IS NOT NULL
     AND billable_from <= $period_end
     AND (active = true OR (deactivated_at IS NOT NULL AND deactivated_at >= $period_start))
   ```
3. If count = 0, return without inserting (no zero-amount invoices).
4. Read `center.price_per_teacher` and `center.currency`.
5. INSERT into `center_invoice` with `ON CONFLICT (center_id, period_month) DO NOTHING` (idempotent).
6. Return the inserted invoice (or existing one if already generated).

## 10. Acceptance criteria — Phase 2 invoice verification

Before rolling out to all centers, verify against 2 hand-picked centers:

| # | Criterion | How to verify |
|---|-----------|---------------|
| 1 | Invoice totals match manual count | Manually count active teachers in the month, compare with generated teacher_count |
| 2 | No teacher with `billable_from > period_end` is billed | Query: `SELECT * FROM admin WHERE center_id=X AND role='teacher' AND billable_from > period_end` should return no rows that appear in the invoice |
| 3 | Deactivated teachers billed iff `deactivated_at >= period_start` (or `deactivated_at IS NULL AND active=true`) | List all inactive teachers, check their `deactivated_at` against period_start |
| 4 | Owners never counted | Query: `SELECT COUNT(*) FROM admin WHERE center_id=X AND role='owner'` and confirm they do not appear in teacher_count |

---

## 11. What TeachHub does NOT do

- TeachHub does **not** invoice students or parents.  
- TeachHub does **not** handle center-to-parent payments (tuition, session fees).  
- TeachHub does **not** store session rates (`classroom.session_rate` and `billing_enabled` are removed).  
- TeachHub does **not** lock out teachers or owners for unpaid invoices.  
- TeachHub does **not** run automated billing crons in Phase 2 (manual trigger only).
